# Workspace Detection

bakar auto-detects the workspace root and BSP family from the current directory. Most commands accept `--workspace` / `-w` to override.

## How detection works

`_workspace_from_cwd()` walks up from the current directory looking for:

1. A `.bakar.toml` marker file (explicit workspace root for generic/BYO projects)
2. The presence of `nxp/` or `ti/` subdirectories (Variscite convention)

If neither is found, the command exits with a helpful message asking you to pass `--workspace` or create a `.bakar.toml` marker.

## BSP families

| Family | Manifest type | Sync tool | Notes |
|--------|--------------|-----------|-------|
| `nxp` | `imx-*.xml` repo manifest | `repo init+sync` | NXP i.MX SoMs |
| `ti` | `processor-sdk-*-config_var<N>.txt` | `oe-layertool populate` | TI Sitara SoMs |
| `bbsetup` | `config/config-upstream.json` (bitbake-setup) | `bitbake-setup init` | Upstream Yocto |
| `generic` | Arbitrary kas YAML | none | Any BSP not matching the above |

## Directory layouts

**NXP workspace** (Variscite):

```text
<workspace>/
  nxp/
    .repo/           - repo tool state
    sources/         - layer checkouts (meta-imx, poky, etc.)
    build/
      conf/          - bblayers.conf, local.conf
      runs/          - build telemetry
    kas-nxp.yml      - generated by bakar gen-kas
```

**TI workspace** (Variscite):

```text
<workspace>/
  ti/
    sources/         - layer checkouts
    build/
      conf/
      runs/
    kas-ti.yml
```

**bitbake-setup workspace**:

```text
<setup-dir>/                 - the directory passed to bitbake-setup init
  config/
    config-upstream.json     - bitbake-setup source of truth
  layers/                    - layer checkouts
  build/
    runs/
  kas-bbsetup.yml            - generated by bakar gen-kas / build
```

**Generic / BYO workspace**:

```text
<dir>/
  my-project.yml     - your kas YAML
  build/
    runs/
  .bakar.toml       - optional marker; presence declares this as the workspace root
```

## BSP family dispatch

When a manifest filename is supplied (`-f`), bakar classifies it:

- `imx-A.B.C-X.Y.Z.xml` -> `nxp`
- `processor-sdk-*-config_var<N>.txt` -> `ti`
- Entries from `~/.config/bakar/vendors.toml` are checked first

When a kas YAML is supplied (positional argument), bakar inspects its content to classify:

- Contains a `meta-imx`, `meta-nxp*`, or `meta-variscite-bsp-imx` layer entry -> `nxp`
- Contains a `meta-ti-bsp`, `meta-ti`, or `meta-variscite-bsp-ti` layer entry -> `ti`
- Otherwise -> `generic`

When neither manifest nor YAML is supplied and the CWD (or `--workspace`) is a bitbake-setup workspace -> `bbsetup`.

## Markers and overrides

Create `.bakar.toml` in any directory to declare it as a workspace root:

```toml
# empty file is sufficient; content is reserved for future use
```

Then any command run from within that directory tree will use it as the workspace root without the `nxp/`/`ti/` subdirectory heuristic.
